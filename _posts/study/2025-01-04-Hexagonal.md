# 🤔 헥사고날 아키텍처란? (Hexagonal Architecture)

헥사고날 아키텍처(Hexagonal Architecture)는 2005년 앨리스터 코버번(Alistair Cockburn)이 제안한 소프트웨어 아키텍처 스타일로, "포트와 어댑터 아키텍처(Ports and Adapters Architecture)"로도 알려져 있습니다. 이 아키텍처의 주요 목적은 애플리케이션의 핵심 비즈니스 로직을 외부 세계로부터 분리하여 유연성과 테스트 가능성을 극대화하는 데 있습니다.

# 헥사고날 아키텍처의 기본 원칙

헥사고날 아키텍처는 애플리케이션을 중심에 두고, 애플리케이션과 외부 환경(사용자 인터페이스, 데이터베이스, 외부 시스템 등)을 연결하는 포트(Ports)와 어댑터(Adapters)를 정의합니다. 이를 통해 시스템의 의존성을 최소화하고, 모듈화를 강화하며, 코드의 변경 용이성을 높입니다.

### 핵심 요소

1. **도메인 중심 설계(Domain-Centric Design):**
   - 애플리케이션의 중심은 도메인 모델과 비즈니스 로직으로 구성됩니다.
   - 도메인 로직은 외부 환경에 의존하지 않고 독립적으로 설계됩니다.
2. **포트(Ports):**
   - 애플리케이션과 외부 환경 간의 인터페이스 역할을 합니다.
   - 주로 애플리케이션이 제공하는 서비스(API)와 외부로부터 데이터를 입력받는 입력 포트를 포함합니다.
3. **어댑터(Adapters):**
   - 포트가 정의한 인터페이스를 구현하여 실제 외부 환경과의 통신을 담당합니다.
   - 예를 들어, 데이터베이스 어댑터, REST API 어댑터, 메시지 큐 어댑터 등이 있습니다.
4. **애플리케이션 코어(Application Core):**
   - 도메인 모델과 애플리케이션 서비스로 구성된 핵심 비즈니스 로직 부분입니다.
   - 외부 기술이나 인프라에 의존하지 않습니다.

# 🫥 헥사고날 아키텍처의 장단점

## 장점 :

1. **유연성:**
   - 외부 환경(예: 데이터베이스, UI, API 등)을 쉽게 교체하거나 확장할 수 있습니다.
2. **테스트 용이성:**
   - 비즈니스 로직을 독립적으로 테스트할 수 있어 단위 테스트와 통합 테스트 작성이 용이합니다.
3. **의존성 역전:**
   - 애플리케이션 코어는 외부 환경에 의존하지 않고, 외부가 코어에 의존하도록 설계됩니다.
4. **유지보수성:**
   - 모듈화가 잘 되어 있어 변경 사항이 국지적으로 발생하며, 코드의 가독성과 유지보수성이 향상됩니다.

## 단점 :

1. **복잡성 증가:**
   - 포트와 어댑터를 정의하고 관리해야 하므로 초기 설계 및 구현 단계에서 복잡성이 증가할 수 있습니다.
2. **학습 곡선:**
   - 헥사고날 아키텍처를 처음 접하는 개발자들에게는 개념과 구조를 이해하는 데 시간이 걸릴 수 있습니다.
3. **초기 투자 비용:**
   - 작은 프로젝트나 단순한 애플리케이션에서는 이 아키텍처가 과도하게 느껴질 수 있으며, 초기 설계 비용이 높아질 수 있습니다.
4. **추가 코드량:**
   - 포트와 어댑터를 구현하면서 코드베이스가 늘어나고, 관리해야 할 코드의 양이 많아질 수 있습니다.
